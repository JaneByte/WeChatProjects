# 项目开发规范

## 协作与输出规范（极简协作原则）

- **默认执行模式**：你说“继续” = 按既定优先级直接做下一项，不重复背景，不额外确认。
- **回复结构固定为三行**：  
  1. 改了什么  
  2. 为什么改  
  3. 下一步  
     （信息够用且最省字）
- **非高风险改动不预告方案**，直接改。  
  高风险情况（会改业务逻辑、会删文件/大段代码、接口/表结构不确定）才提前说明。
- **只贴“改动片段 + 文件路径”**，不贴整个文件。
- **不重复复述历史结论**，除非你明确要求“总结/复盘”。
- **注释策略**：只保留必要注释，不新增解释性重复注释。
- **清理任务优先“最小 diff”**（局部替换），避免整文件重写。
- **验证输出只报结论**：通过/失败 + 1 行关键信息，不贴冗长日志。
- **你说“不要复查”时**，停止额外扫描和扩展建议。
- **默认不主动造示例数据、不扩展重构**，除非你明确要求。
- **统一短指令约定**：  
  - `继续` = 下一项  
  - `跳过` = 跳过当前项  
  - `只修bug` = 不做样式/重构  
  - `仅最小改动` = 不改结构

---

## 数据库规范

- 数据库表名与字段名统一使用 `snake_case`；Java 实体/DTO/VO 统一使用 `camelCase`
- 新增业务表默认使用 `InnoDB` 引擎，禁止新建 `MyISAM` 表
- 所有跨表关系必须声明外键或等价约束策略（至少在设计文档中明确）
- 涉及保留字（如 `order`、`user`）时，SQL 中必须使用反引号包裹；新表命名尽量规避保留字

---

## 后端规范

- 所有后端查询必须通过 Mapper 层，禁止在 Controller/Service 中拼接原生 SQL 字符串
- MyBatis 参数必须使用 `#{}` 占位，禁止使用 `${}` 直接拼接用户输入
- API 返回统一使用 `ApiResponse<T>`（包含 `code`/`message`/`data`/`timestamp`），禁止返回 `Map` 或裸 `String`
- 必须启用 `@RestControllerAdvice` 全局异常处理，Controller 层不分散写 `try-catch`

---

## 前端规范

- 所有网络请求统一走 `utils/request.js`，页面中禁止直接使用 `wx.request` 或 `fetch`
- API 地址仅允许从 `config` 文件读取，禁止在页面中硬编码 `baseUrl` 或第三方接口地址
- 页面创建流程固定：创建四件套（`.js` / `.wxml` / `.scss` / `.json`）→ 注册路由 → 接入 `request` 层 → 补齐异常态/空态
- 样式默认使用 `rpx`；仅在 `media query` / 滤镜等必要场景可使用 `px`，且需注释说明；禁止使用 `!important`

---

## 通用规范

- **编码要求**：所有生成的代码文件（`.js`/`.json`/`.wxml`/`.wxss`/`.scss`）必须使用 **UTF-8 无 BOM** 编码，禁止使用 GBK/GB2312 等编码。
- **输出内容必须直接使用 Unicode 字符（如中文）**，禁止输出转义序列（如 `\u4e2d\u6587`）。
- **禁止在代码中插入任何编码声明注释**（如 `// -*- coding: utf-8 -*-`），避免干扰编辑器自动检测。
- **禁止行为**：严禁为了修改少量代码而**删除整个文件再重写**。只能使用“局部替换”或“按行修改”的方式。
- **最小 diff 原则**：每次只输出**改动部分的代码片段**，并标注文件路径和行号（如 `pages/index/index.js` 第 45-52 行）。
- **如果必须大改**：必须先向我说明理由，等我确认后才能输出完整文件。

## 编码与乱码防护规则

### 核心原则
1. 所有新建/修改的前端文本文件（`.js`/`.wxml`/`.wxss`/`.scss`/`.json`/`.md`）必须以 **UTF-8（无 BOM）** 保存。  
   *（理由：统一编码是防乱码第一原则）*
2. 中文字符串必须保证引号成对闭合；字符串统一使用单引号（`'`）。  
   *（理由：降低编码异常导致的解析错误风险）*

### 操作规范
3. 每次修改含中文内容后，必须执行一次**乱码特征扫描**（如查找 `鍔/鏆/璇/�/楼` 等异常字符），并在回复中报告扫描结果。  
   *（理由：主动发现并拦截乱码）*
4. 每次修改 `.js` 文件后，必须执行语法检查（`node --check`）并报告结果。  
   *（理由：乱码常导致引号断裂和语法报错）*
5. 若发现文件已出现乱码且结构损坏（如标签闭合/引号丢失），**优先整文件重写**为 UTF-8 正常文本，不做局部猜测修补。  
   *（理由：局部修补容易遗留隐患）*

### 禁止项
6. **禁止**对中文文案做“无上下文批量盲替换”；涉及中文替换时必须先定位文件与上下文。  
   *（理由：避免误伤正常文本、引号和模板结构）*
7. **禁止**将乱码字符写入最终代码（如：`鍔`、`鏆`、`璇`、`�`、`楼` 作为货币符号）。  
   *（理由：明确不可接受输出，避免反复污染）*
8. 价格符号统一使用 `¥`，**禁止**使用异常替代符号（如 `楼`）。  
   *（理由：该问题在本项目出现频率高，需硬性约束）*

### 特殊场景
9. 使用 PowerShell 写文件时，必须显式指定 `-Encoding UTF8`（无 BOM）。  
   *（理由：避免系统默认编码造成二次乱码）*

### 修复结果要求
10. 若本轮出现乱码修复，回复中必须包含：**修复文件清单 + 语法检查结果 + 乱码终检结果**。  
    *（理由：保证修复过程可追溯、可验收）*
